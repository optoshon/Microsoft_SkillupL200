Module 5 – From LLM Calls to Agents & Workflows

In this module, we zoom out from "just calling a model" and learn how to use LLMs in the right way for the right problem – as simple prompt apps, as deterministic workflows, and as agentic systems that can plan, act, and self-evaluate.

What this module helps you do

Think of LLMs as different "colleagues" you can hire: one who just answers questions, one who follows a fixed checklist, and one who can improvise, use tools, and ask for second opinions. Module 5 is about knowing which colleague to call.

  • Recognise when you just need a simple prompt app vs a multi-step workflow vs an agent.
  • Apply these patterns to a concrete scenario: a Proposal Assistant that drafts client-ready proposals.
  • Start with a pure SDK implementation, then see how the same ideas map into LangChain as homework.

The goal is not to memorise APIs, but to build a mental model you can reuse across Azure AI, LangChain, Prompt Flow, n8n, or any other orchestration tool.

Patterns & Vocabulary: From Simple Calls to Agentic Systems

LLM Usage Patterns Workflow vs Agent RAG in the Mix

This section introduces the terminology you'll use for the rest of Module 5: simple LLM calls, chat with memory, prompt apps, fixed workflows, agents, agentic systems, and where RAG fits into all of those.

Pattern What it feels like Typical use Where you see it in Module 5
LLM call Ask once, get one answer. Single-shot questions, simple text transforms, quick experiments. Intro notebook – basic SDK call.
Chat + memory A running conversation where the model "remembers" what you said (because we resend it). Assistants, chatbots, support bots that need context from previous turns. Intro notebook – manual message history.

Complex, long-running flows: multi-stage review, multi-team workflows, orchestration across tools and services. RAG (Retrieve → Read → Respond) The model "looks something up" before answering. Question-answering over documents, policy-aware assistants, knowledge bots. Mostly covered in earlier modules; here we show where RAG could be plugged into workflows or agents.

Analogy: Kitchen roles vs recipes

Think of these patterns as different ways of running a kitchen:

  • LLM call: You ask the chef one question ("How do I boil an egg?") and get one answer.
  • Prompt app: You have a recipe card where you fill in a few blanks (dish name, number of people, spice level).
  • Workflow: You follow a fixed recipe step-by-step, every time.
  • Agent: You trust the chef to decide what to do next – check the pantry, adjust the recipe, ask a sous-chef for help.
  • Agentic system: A whole kitchen brigade with multiple chefs, runners, and reviewers, all coordinating.
  • RAG: Giving the chef access to a well-organised pantry and cookbook shelf.

Use Case 1 – Proposal Assistant: One Scenario, Many Patterns

Business Scenario Workflow & Agent Views SDK Implementation

We're going to use a single, realistic scenario—a Proposal Assistant—to connect all the patterns. The assistant helps create client-ready proposals for GenAI/Azure solutions based on a high-level requirement or RFP-style description.

What the Proposal Assistant needs to do

Internal teams or sales teams often need to respond quickly to GenAI opportunity requests. They have rough context and a few bullet points, but they need a structured, client-ready proposal that aligns with Azure and internal standards.

Inputs

  • High-level requirement or RFP-style text
  • Optional client context (industry, region, existing systems)
  • Internal constraints (must use Azure OpenAI, security/compliance notes)

Outputs

  • A structured proposal with sections like:
    • Introduction / Executive summary
    • Business context & problem statement
    • Proposed GenAI/Azure solution overview
    • High-level architecture
    • Implementation phases and rough timeline
    • Risks, assumptions, and next steps

Constraints

  • Should be consistent in structure across proposals
  • Should be custom to the requirement, not just a generic template
  • Should be easy to tweak manually after generation

Why this use case? Because it is familiar to many teams (responding to RFPs / proposals), and because it can be implemented with both patterns: as a simple multi-step workflow, and as an agentic pattern where the system plans, drafts, and evaluates.

Workflow Solution – Deterministic Pipeline with the SDK

A workflow is a fixed pipeline of steps that we define in code. The LLM fills in content at each step, but it does not decide which step comes next. For the Proposal Assistant, we use a workflow to go from requirement → outline → section drafts → final proposal in a predictable way.

SDK-only Implementation Deterministic Workflow No Tool Selection

How the workflow solution behaves

In the workflow approach, we treat the proposal generation process like an assembly line. Every input runs through the same ordered steps: understand the requirement, produce an outline, draft each section, then optionally refine style and tone. The code controls the sequence; the LLM focuses on filling in the content.

Workflow Steps

  1. Step 1 – Understand and structure the requirement
    Optionally clean the requirement text and highlight key objectives and constraints.
  2. Step 2 – Generate an outline
    Ask the model to propose a structured outline for the proposal (section headings, rough ordering).
  3. Step 3 – Draft each section
    For each section in the outline, call the model to generate a detailed draft, one section at a time.
  4. Step 4 – Refine style and tone (optional)
    Optionally run a final pass to make the proposal more concise, executive-friendly, or aligned with a chosen tone.
  5. Step 5 – Assemble final proposal
    Join sections into a single document that can be reviewed, edited, or directly pasted into a deck or document template.
Stage What happens Input Output
Requirement intake & normalisation We capture the requirement text and optionally clean or lightly rephrase it. Raw requirement / RFP text from the team. Normalised requirement text used in later prompts.
Generate outline The model proposes a structured outline for the proposal (section headings and order). Normalised requirement text. Outline with sections like Intro, Context, Solution, Architecture, Phases, Risks.
Draft sections We iterate over the outline and ask the model to draft each section. Requirement text + outline (and current section name). Draft text for each proposal section.
Refine style & tone Run a final pass to adjust tone (e.g., executive summary style, concise, formal). Full draft proposal text. Refined proposal with consistent voice and tone.
Assemble & handoff Join all sections, apply any lightweight formatting, and hand over for human review. Refined sections and proposal text. Client-ready draft proposal to be reviewed/edited by a human.

When to start with a workflow first

For many real projects, starting with a simple workflow is the safest choice:

  • Your overall process is already well understood (e.g., you know the standard sections in every proposal)
  • You need predictable token cost and latency per request
  • You want the orchestration logic to stay simple and debuggable (each step can be logged and tested separately)
  • You are comfortable asking humans to do the final quality check on the output
  • You might add agentic behaviour later, but you don't want to start there

This same workflow will be extended in the next section into an agentic variant for scenarios where we want the system to plan and self-evaluate.

Agentic Solution – Planner, Drafter, Evaluator Loop

In the agentic version of the Proposal Assistant, we keep the same overall goal but give the system more responsibility. Instead of a fixed pipeline, we split the work into roles (planner, drafter, evaluator) and let the model's evaluation decide whether to stop or iterate. The control flow becomes dynamic: the assistant can generate, critique, and refine proposals.

Agentic Behaviour Planner / Drafter / Evaluator Dynamic Control Flow

Three roles in the agentic proposal assistant

To make the Proposal Assistant more self-directed, we break the work into three specialised roles. Each role uses the LLM differently and passes its output to the next one. Together, they behave like a tiny team: one plans, one drafts, one reviews.

Planner agent

  • Reads the requirement and extracts key goals, constraints, and scope
  • Proposes a structured plan or outline for the proposal: sections, ordering, and what each part should cover
  • Ensures the outline reflects both business outcomes and technical expectations (e.g., Azure, GenAI, security)

Drafter agent

  • Takes the requirement and the planner's outline
  • Generates a full proposal draft, section by section, following the plan
  • Aims for clarity, enough detail for discussion, and alignment with the defined structure

Evaluator agent (LLM-as-judge)

  • Reads the requirement and the current draft proposal
  • Scores the draft along dimensions like relevance, structure, technical soundness, and tone
  • Returns a score and specific feedback: what to improve, what is missing, what is confusing

An orchestrator function glues these roles together, deciding whether to accept the draft or loop back and ask the drafter to try again using the evaluator's feedback.

flowchart TD A[User Requirement] --> B[Planner] B --> C[Drafter] C --> D[Evaluator] D -->|Score too low
Needs revision| C D -->|Score good enough
Approved| E[Final Proposal] style E fill:#E8F5E9,stroke:#107C10 style D fill:#FFF8E1,stroke:#FFB900

The evaluator's score and feedback control whether the loop repeats, which is what makes this more "agentic" than the fixed workflow in the previous section.

Trade-offs of the agentic approach

  • You can often get higher quality or more robust proposals by allowing iteration and self-evaluation
  • But you also pay more in tokens and latency, because you may generate multiple drafts and evaluations
  • Debugging is more complex: you need to inspect planner outputs, drafts, and evaluations to understand what went wrong
  • It can be harder to guarantee strict determinism: the loop depends on model-generated scores and feedback
  • For many teams, a good strategy is to start with a simple workflow, then introduce agentic elements only where they clearly add value (e.g., critical proposals or complex deals)

Next, we'll put the workflow and agentic solutions side by side and discuss when to choose one over the other.

Workflow vs Agent – How to Choose

We've seen the Proposal Assistant implemented both as a deterministic workflow and as an agentic loop with planner, drafter, and evaluator. This section summarises the trade-offs and gives you a simple way to think about which pattern to start with in your own projects.

Pattern Comparison Trade-offs Decision Hints
Aspect Workflow-first Agent / Hybrid
Process & orchestration Orchestration is defined entirely in code: the steps and order are fixed. The model's outputs (e.g., evaluation scores) influence control flow and iteration.
When steps are known Ideal when you already know the sequence and structure of steps (like a standard proposal template). Useful when the path can vary by case, or you want the system to decide what to do next.
Cost & latency More predictable token usage and latency; easy to estimate per run. Often higher and more variable, because of multiple drafts and evaluations.
Correctness & quality Quality depends mostly on good prompts per step and human review. Can improve quality through self-evaluation and iterative refinement, at the cost of more complexity.
Debuggability Easier to trace: log each step, inspect intermediate outputs. More moving parts: need to inspect plan, drafts, evaluations, and loop decisions.
Implementation complexity Simple to implement and maintain; good baseline for most teams. More complex orchestration, more prompts to manage, more to explain to stakeholders.
Good starting point? Great default starting point for new use cases. Best reserved for high-value or complex scenarios where extra quality justifies extra cost.

A quick way to think about the choice

Instead of a strict formula, you can ask yourself a few questions on a 1–5 scale and see where you land:

  • How critical is correctness for this use case?
    1 = "rough ideas are fine, human will fully rewrite" | 5 = "mistakes are very costly or embarrassing"
  • How dynamic is the path to the answer?
    1 = "we always follow the same steps" | 5 = "steps change a lot depending on the situation"
  • How tightly do we need to control cost and latency?
    1 = "we can afford multiple iterations" | 5 = "we have strict limits per request"
  • How much implementation complexity can the team handle today?
    1 = "keep it dead simple" | 5 = "we're comfortable with more moving parts"

Rule of thumb: If correctness and dynamism scores are low, start with a workflow. If both are high and you can tolerate cost/complexity, consider adding agentic elements (like a planner/evaluator loop). You can always start with a workflow and gradually "upgrade" to a hybrid approach where it makes sense.

Next Steps & Learning Paths

Different people will use Module 5 differently. This section suggests a few learning paths depending on whether you're more hands-on with code, more focused on architecture, or more focused on evaluation and business impact.

Choose Your Path Role-based Guidance Beyond This Module

If you're a hands-on developer…

You like to learn by building, running, and tweaking code.

  • Make sure you can run:
    • The intro notebook (Module 5 patterns)
    • The Proposal Assistant workflow solution (SDK)

  • Pick one internal use case from your team and sketch:

    • A possible agentic extension (e.g., add an evaluator loop)
  • Experiment with adding:
    • A small RAG step (e.g., pulling from a tiny document set)
    • Or an extra tool (like a calculator or a knowledge lookup) in your agentic variant

If you're an architect or tech lead…

You care about patterns, governance, and making good choices across many projects.

  • Use the Proposal Assistant as a template for:
    • How to document inputs, outputs, and constraints
    • How to express a workflow vs an agentic design for the same problem
  • For 2–3 real use cases in your organisation:
    • Sketch the workflow view (fixed steps)
    • Sketch the agentic view (where planner/evaluator roles might sit)
    • Decide where you would start (workflow vs agent vs hybrid) and why
  • Define a small internal checklist:
    • Questions to ask when choosing between workflow and agent (correctness, cost, complexity, etc.)
  • Work with devs to:
    • Standardise patterns (e.g., "most use cases start as workflows, we add evaluators only in these conditions")

If you're a PM, evaluator, or business owner…

You care about impact, safety, and how to evaluate these systems.

  • Focus on understanding:
    • What a workflow is vs an agent
    • Where human review fits into each
  • For your own use cases, ask:
    • "What does a 'good enough' answer look like?"
    • "Where is human oversight mandatory?"
  • Collaborate with a dev/architect to:
    • Define evaluation criteria for proposals (relevance, structure, tone, risk coverage)
    • Decide whether an extra evaluator loop is worth the extra cost/latency
  • Use the Proposal Assistant examples as:
    • A template for how to describe requirements and outputs
    • A starting point for writing acceptance criteria for other GenAI features

You don't have to do everything at once. Pick the path that fits your role, and focus on internalising the patterns. Frameworks and tools will change; the way you think about workflows, agents, and evaluation will remain useful.
